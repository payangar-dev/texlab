# Implementation Plan: Undo/Redo System (Snapshot-Based)

**Branch**: `004-undo-redo-system` | **Date**: 2026-03-29 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-undo-redo-system/spec.md`

## Summary

Implement a snapshot-based undo/redo system for texture editing. Each user action (draw stroke, layer add/remove/reorder, property change) captures a full-state snapshot of the layer stack before the mutation. An `UndoManager` (domain) maintains undo/redo stacks with a max depth of 100. An `EditorService` (use cases) orchestrates all undoable operations, ensuring every mutation is recorded. Backend only — no frontend integration in this feature.

## Technical Context

**Language/Version**: Rust ≥ 1.77
**Primary Dependencies**: None (domain layer uses std only). Use cases layer imports only from domain.
**Storage**: In-memory only (undo history is not persisted to disk)
**Testing**: `cargo test` — unit tests with in-memory data, no I/O
**Target Platform**: Desktop (Windows, macOS, Linux via Tauri)
**Project Type**: Desktop application (Tauri v2)
**Performance Goals**: Undo/redo operations < 1ms for 64x64 textures with 10 layers
**Constraints**: Memory budget ~16MB worst case (64x64, 10 layers, 100 snapshots). Trivial for desktop.
**Scale/Scope**: Single-texture editing, max 100 undo steps
